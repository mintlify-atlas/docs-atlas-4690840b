## Welcome Contributors!

flowState CLI is built with Go and follows strict Test-Driven Development (TDD) practices. This guide will help you get started with contributing.

### Clone & Build

<Steps>
  <Step title="Clone the repository">
    ```bash
    git clone https://github.com/Jericoz-JC/flowState-CLI
    cd flowState-CLI
    ```
  </Step>
  
  <Step title="Download dependencies">
    ```bash
    go mod download
    ```
    
    This fetches all Go modules defined in `go.mod`:
    - Bubble Tea (TUI framework)
    - Lip Gloss (styling)
    - SQLite (database)
    - ~30 transitive dependencies
  </Step>
  
  <Step title="Build the binary">
    ```bash
    go build -o flowstate ./cmd/flowState
    ```
    
    This creates a `flowstate` binary in the current directory.
  </Step>
  
  <Step title="Run the application">
    ```bash
    ./flowstate
    ```
    
    First run will initialize the SQLite database and download the embedding model (~90MB).
  </Step>
</Steps>

## Test-Driven Development (TDD)

<Warning>
**MANDATORY:** All new features MUST follow TDD. No exceptions.
</Warning>

### The RED-GREEN-REFACTOR Cycle

<Steps>
  <Step title="🔴 RED - Write a failing test">
    ```go
    func TestParseWikilinks(t *testing.T) {
        input := "see [[Note Title]]"
        result := parseWikilinks(input)
        expected := []string{"Note Title"}
        
        if !reflect.DeepEqual(result, expected) {
            t.Errorf("got %v, want %v", result, expected)
        }
    }
    ```
    
    Run the test and **verify it fails for the RIGHT reason** (missing function, not syntax error).
  </Step>
  
  <Step title="✅ GREEN - Implement minimal code to pass">
    ```go
    func parseWikilinks(text string) []string {
        re := regexp.MustCompile(`\[\[([^\]]+)\]\]`)
        matches := re.FindAllStringSubmatch(text, -1)
        
        var links []string
        for _, match := range matches {
            links = append(links, match[1])
        }
        return links
    }
    ```
    
    Run the test again and verify it passes. Run **ALL** tests to ensure nothing broke.
  </Step>
  
  <Step title="🔄 REFACTOR - Clean up while maintaining green">
    ```go
    // Extract regex to package constant
    var wikilinkRegex = regexp.MustCompile(`\[\[([^\]]+)\]\]`)
    
    func parseWikilinks(text string) []string {
        matches := wikilinkRegex.FindAllStringSubmatch(text, -1)
        // ... rest of implementation
    }
    ```
    
    Tests still pass after refactoring!
  </Step>
</Steps>

### Table-Driven Tests (Preferred Pattern)

Go's idiomatic testing pattern:

```go
func TestExtractTags(t *testing.T) {
    tests := []struct {
        name     string
        input    string
        expected []string
    }{
        {
            name:     "empty string",
            input:    "",
            expected: nil,
        },
        {
            name:     "single hashtag",
            input:    "hello #world",
            expected: []string{"world"},
        },
        {
            name:     "multiple tags",
            input:    "#go #programming @mentions",
            expected: []string{"go", "programming", "mentions"},
        },
        {
            name:     "tags in title and body",
            input:    "#title\n\nBody with #tag",
            expected: []string{"title", "tag"},
        },
    }
    
    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            result := extractTags(tt.input)
            if !reflect.DeepEqual(result, tt.expected) {
                t.Errorf("got %v, want %v", result, tt.expected)
            }
        })
    }
}
```

**Benefits:**
- Easy to add new test cases
- Clear test names with `t.Run()`
- Isolated test execution
- Comprehensive edge case coverage

### Red Flags - STOP and RESTART

<Warning>
If any of these occur, you're doing it wrong:

- ❌ You wrote production code before the test
- ❌ The test passed immediately (didn't test anything new)
- ❌ You can't explain why the test should fail
- ❌ You skipped running tests after implementation
</Warning>

## Code Style Guidelines

### Go Formatting

```bash
# Format all code (automatically fixes style)
go fmt ./...

# Run linter (requires golangci-lint)
golangci-lint run
```

### Naming Conventions

| Type | Convention | Example |
|------|------------|----------|
| **Packages** | Lowercase, single word | `screens`, `models`, `storage` |
| **Types** | PascalCase | `NotesModel`, `TodoItem` |
| **Functions** | camelCase (private), PascalCase (public) | `parseWikilinks()`, `GetNotes()` |
| **Constants** | PascalCase or SCREAMING_SNAKE_CASE | `MaxItemsPerPage`, `DEFAULT_TIMEOUT` |
| **Variables** | camelCase | `currentNote`, `isEditing` |

### Bubble Tea Patterns

All screens follow this structure:

```go
package screens

import (
    tea "github.com/charmbracelet/bubbletea"
)

// Message types (private)
type notesLoadedMsg struct {
    notes []models.Note
    err   error
}

// Model (exported)
type NotesModel struct {
    // State fields
    notes  []models.Note
    cursor int
    
    // Dependencies (injected)
    store storage.Store
}

// Constructor
func NewNotesModel(store storage.Store) NotesModel {
    return NotesModel{
        store: store,
    }
}

// Bubble Tea interface
func (m NotesModel) Init() tea.Cmd {
    return nil  // or return a command
}

func (m NotesModel) Update(msg tea.Msg) (tea.Model, tea.Cmd) {
    // Handle messages
    return m, nil
}

func (m NotesModel) View() string {
    // Render UI
    return "..."
}
```

### Error Handling

```go
// Return errors, don't panic
func loadNotes(store storage.Store) ([]models.Note, error) {
    notes, err := store.GetNotes()
    if err != nil {
        return nil, fmt.Errorf("failed to load notes: %w", err)
    }
    return notes, nil
}

// Handle errors in Update()
case notesLoadedMsg:
    if msg.err != nil {
        m.errorMessage = msg.err.Error()
        return m, nil
    }
    m.notes = msg.notes
    return m, nil
```

### Hot Reload with Air


### Manual Build Loop

```bash

while true; do
    go build -o flowstate ./cmd/flowState

```

## Systematic Debugging

When fixing bugs, follow this process:

<Steps>
  <Step title="Root Cause Investigation">
    - Read the error message carefully
    - Reproduce the issue consistently
    - Trace data flow through the code
    - Use `log.Printf()` to inspect values
  </Step>
  
  <Step title="Pattern Analysis">
    - Find working examples of similar code
    - Identify differences between working and broken code
    - Check for common pitfalls (nil pointers, empty slices, etc.)
  </Step>
  
  <Step title="Hypothesis Testing">
    - Make ONE change at a time
    - Test after each change
    - Document what you tried
  </Step>
  
  <Step title="Implementation">
    - Write a failing test that reproduces the bug
    - Implement the fix
    - Verify the test passes
    - Run ALL tests to ensure no regressions
  </Step>
</Steps>

<Warning>
**Never propose fixes without understanding the root cause first.**
</Warning>

## Pull Request Process

<Steps>
  <Step title="Create a feature branch">
    ```bash
    git checkout -b feature/your-feature-name
    ```
  </Step>
  
  <Step title="Write tests first (TDD)">
    Follow the RED-GREEN-REFACTOR cycle.
  </Step>
  
  <Step title="Implement the feature">
    Keep commits atomic and descriptive.
  </Step>
  
  <Step title="Run the full test suite">
    ```bash
    go test ./...
    go fmt ./...
    ```
  </Step>
  
  <Step title="Update documentation">
    If your feature is user-facing, update:
    - README.md (user guide)
    - CLAUDE.md (development plan)
  </Step>
  
  <Step title="Push and create PR">
    ```bash
    git push origin feature/your-feature-name
    ```
    
    Open a PR on GitHub with:
    - Clear description of changes
    - Link to related issues
    - Screenshots/GIFs for UI changes
  </Step>
</Steps>
