## Overview

flowState's semantic search finds notes based on **meaning**, not just keywords. Search using natural language queries and get results ranked by relevance, even when the exact words don't appear in your notes.

<Info>
**100% Local & Private**: All search happens on your machine using a local ONNX model. Your notes never leave your device.
</Info>

## How It Works

### Semantic Embeddings

flowState uses the `all-MiniLM-L6-v2` transformer model to convert text into numerical vectors:

1. **Text → Vector**: Your notes are converted to 384-dimensional embeddings
2. **Query → Vector**: Your search query is converted to the same embedding space
3. **Similarity**: Results are ranked by cosine similarity between vectors
4. **Ranking**: Most similar notes appear first

<Note>
Embeddings capture semantic meaning - "project meeting" will match notes about "team standup" even without shared keywords.
</Note>

## The Model

### all-MiniLM-L6-v2

flowState uses this proven sentence-transformer model:

- **Model size**: ~90MB (downloaded on first run)
- **Dimensions**: 384 (balanced between accuracy and speed)
- **Architecture**: Sentence-BERT based on MiniLM
- **Source**: HuggingFace `sentence-transformers/all-MiniLM-L6-v2-onnx`
- **Format**: ONNX for cross-platform inference

<Tip>
The model downloads automatically on first search. Expect a one-time ~90MB download.
</Tip>

## Understanding Results

### Similarity Scores

Each result shows a score indicating match quality:

- **0.8 - 1.0**: Excellent match (highly relevant)
- **0.6 - 0.8**: Good match (relevant)
- **0.4 - 0.6**: Fair match (somewhat relevant)
- **< 0.4**: Weak match (might not be relevant)

<Info>
Scores are calculated using cosine similarity between query and note embeddings. Higher scores mean the semantic meaning is more similar.
</Info>

### Result Ranking

Results are sorted by descending score:

1. Highest similarity first
2. Equal scores maintain database order (usually newest first)
3. Low-scoring results are still shown (you decide the cutoff)

### Result Display

Each result shows:

- **Score**: Similarity score as `[0.XX]`
- **Preview**: First line of the note (usually the title)
- **Selection**: Highlighted with cyan background when selected

## Storage & Indexing

### SQLite Vector Storage

Embeddings are stored in the `note_vectors` table:

```sql
CREATE TABLE note_vectors (
    note_id INTEGER PRIMARY KEY,
    embedding BLOB NOT NULL,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

- **384 floats per note** (1536 bytes as blob)
- **Indexed by note_id** for fast lookup
- **Cascade delete** when notes are removed

### Automatic Indexing

Notes are indexed automatically:

- **On create**: New notes are indexed when saved
- **On update**: Edited notes are re-indexed
- **On delete**: Embeddings are removed when notes are deleted

<Note>
Indexing happens in the background. Large notes may take a moment to index, but you can continue working.
</Note>

### Index Content

Each note's embedding is generated from:

- **Title**: Note title (full weight)
- **Body**: Note body (combined with title)
- **Combined**: `title + "\n" + body` for maximum context

<Tip>
Include context in your note titles for better search results. "Project X Meeting Notes - Feb 28" is more searchable than just "Meeting Notes".
</Tip>

## Privacy & Security

### 100% Local

Semantic search runs entirely on your machine:

- ✅ **Model**: Downloaded once and stored locally
- ✅ **Embeddings**: Generated on your device
- ✅ **Search**: All computation happens locally
- ✅ **Storage**: SQLite database on your filesystem
- ❌ **No cloud**: Zero API calls to external services
- ❌ **No telemetry**: Your searches are never logged or sent anywhere

<Info>
Your notes, queries, and embeddings never leave your machine. flowState is privacy-first by design.
</Info>

### Data Storage

Your semantic search data is stored in:

```
~/.config/flowstate/flowstate.db          # SQLite database
~/.config/flowstate/models/               # ONNX model directory
  └── all-MiniLM-L6-v2/
      └── model.onnx                       # ~90MB model file
```

## Performance

### Speed

- **First query**: 1-2 seconds (model loading)
- **Subsequent queries**: < 500ms for 100 notes
- **Indexing**: < 100ms per note on save

<Note>
Performance depends on your hardware. Faster CPUs and SSDs improve search speed.
</Note>

### Scalability

Semantic search scales well:

- **100 notes**: Instant search
- **1,000 notes**: < 1 second
- **10,000 notes**: A few seconds

### Resource Usage

- **RAM**: ~200MB for model + embeddings
- **Storage**: 1.5KB per note (384 × 4 bytes)
- **CPU**: Moderate during search, minimal when idle

### Writing Searchable Notes

1. **Descriptive titles**: Include context in note titles
2. **Full sentences**: Write notes in natural language
3. **Include context**: Add who/what/when/where/why
4. **Use synonyms**: Include alternative terms in notes
5. **Link related notes**: Connections improve context

### Query Strategies

1. **Be specific**: Narrow queries get better results than broad ones
2. **Use questions**: "How do I deploy?" finds instructional notes
3. **Include context**: "Q4 project planning" vs. just "planning"
4. **Try variations**: Rephrase queries if results aren't relevant
5. **Check scores**: Scores below 0.4 might not be useful

## Troubleshooting

<AccordionGroup>
  <Accordion title="Model download fails">
    **Symptoms**: Error message about model download on first search
    
    **Solutions**:
    - Check internet connection (one-time 90MB download)
    - Verify disk space (~100MB free needed)
    - Check firewall settings (allow flowstate to access HuggingFace)
    - Try again - download may have been interrupted
    
    **Manual download**:
    ```bash
    # Download from HuggingFace manually
    wget https://huggingface.co/sentence-transformers/all-MiniLM-L6-v2-onnx/resolve/main/model.onnx
    mv model.onnx ~/.config/flowstate/models/all-MiniLM-L6-v2/
    ```
  </Accordion>
  
  <Accordion title="Search returns no results">
    **Possible causes**:
    - No notes created yet (create notes first)
    - Notes not indexed (wait a moment after creating notes)
    - Query too specific (try broader terms)
    - Model not loaded (check for error messages)
    
    **Try**:
    - Create or edit a note to trigger indexing
    - Use broader, simpler queries
    - Restart flowstate to reload the model
  </Accordion>
  
  <Accordion title="Search is slow">
    **Expected on first query**: Model loading takes 1-2 seconds
    
    **If consistently slow**:
    - Check CPU usage (other apps competing for resources)
    - Verify SSD/HDD performance
    - Restart flowstate to clear any memory issues
    - For very large note collections (10,000+), search may take a few seconds
  </Accordion>
  
  <Accordion title="Results don't match my query">
    **Remember**: Semantic search finds meaning, not keywords
    
    **Low scores (< 0.4)**: Probably not relevant - ignore them
    
    **Unexpected results**: The model might see connections you don't
    - Check what words/concepts overlap
    - Rephrase your query more specifically
    - Use exact keyword search if you need literal matches (future feature)
  </Accordion>
</AccordionGroup>

## Technical Details

### Embedding Generation

The flow for creating embeddings:

```
Note (title + body)
  ↓
Tokenization (split into tokens)
  ↓
ONNX Model (all-MiniLM-L6-v2)
  ↓
384-dimensional vector
  ↓
Normalized (unit length)
  ↓
Stored in SQLite (note_vectors.embedding)
```

### Similarity Calculation

Cosine similarity between query and note embeddings:

```
similarity = dot(query_embedding, note_embedding) / 
             (||query_embedding|| × ||note_embedding||)
```

- **Range**: -1.0 to 1.0 (normalized to 0.0 to 1.0 for display)
- **1.0**: Identical meaning
- **0.0**: Completely unrelated
- **Values < 0**: Opposite meaning (rare in practice)
